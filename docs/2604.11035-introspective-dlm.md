## 2. 核心贡献概述

- 中文版：本文把 DLM 与 AR 模型的质量差距归因于缺少 introspective consistency，即模型不能稳定认可自己已经生成的 token；在此基础上提出 I-DLM，用 logit-shifted causal training 把预训练 AR 模型转成可并行解码的 DLM，再用 Introspective Strided Decoding (ISD) 在同一次 forward 中生成新 token 并验证旧 token。系统层面，I-DLM 复用 SGLang / AR serving stack，并通过 stationary-batch scheduler、CUDA graph、paged-only attention 和 Triton fused verify，把理论并行性转成高并发吞吐收益。
- 英文版（abstract）：作者写道，"We define the introspective acceptance rate"，并指出 "I-DLM uses a novel introspective strided decoding (ISD) algorithm"。这两句分别概括了本文的问题刻画和方法入口。

## 3. 一句话精髓（写给外行）

它让扩散语言模型一边往前写，一边回头问自己“刚才写的这几个字我还认不认”，认得就大步前进，认不准就当场修正。

## 4. 学术贡献清单

### 4.1 动机的发现与阐明

作者把既有 DLM 的核心问题从“并行解码还不够快”重新定位为“生成与自我验证不一致”。他们定义 introspective acceptance rate：

$$
\alpha = \frac{1}{L}\sum_k \min\left(1, \frac{p_k(x_k)}{q_k(x_k)}\right),
$$

其中 $q_k$ 是模型生成 token $x_k$ 时的 proposal distribution，$p_k$ 是把已生成 token 显示出来后，模型在 causal anchor 位置给出的 next-token distribution。AR 模型由于训练和推理都使用 causal mask + logit shift，天然有 $p=q$；而 SDAR、LLaDA 2.0-flash 这类 DLM 的 $p$ 和 $q$ 明显分离。论文报告 SDAR (8B) 的 introspective acceptance rate 为 0.699，LLaDA 2.0-flash (8B) 为 0.568，I-DLM 接近 AR。

这个指标的价值在于，它把 DLM 的长链推理失败解释为局部 token 自洽性不足：一个 token 生成时看似高置信，一旦成为可见上下文，模型重新评估时却未必认可它。对数学和代码这类长链任务，这种微小不一致会沿推理链放大。

### 4.2 方法的创新

- Introspective-consistency training：把输入构造成 $[x_t \mid x_0]$，其中 $x_t$ 是全 MASK 序列，$x_0$ 是 clean reference；在 masked region 和 clean region 都使用 strict causal attention 与 logit shift。
- Auto-balanced loss：分别计算 masked pathway 和 clean introspection pathway 的 next-token cross entropy，再动态按 loss ratio 平衡 clean loss，避免更难的 masked prediction 压过验证能力。
- ISD：每步同时 proposal 与 introspection，用 $p/q$ acceptance 决定接受多少 token；接受率高时 stride 自动变长，困难位置则退化到 AR-like 安全路径。
- R-ISD / gated LoRA：在 MASK proposal 位置启用 LoRA residual，在 clean introspection 位置只用 base weights，使 causal anchor $p$ 与原 AR base bit-for-bit 一致。
- Serving stack：将 ISD 映射到 SGLang 的 extend mode，复用 paged KV cache、continuous batching、tensor parallelism，再用 stationary-batch scheduling 和 Triton fused verification 减少 CPU/GPU 同步与 kernel launch。

### 4.3 实验上的核心结论

- 质量：I-DLM-8B 在 15 个 benchmark 上超过已有 DLM，且接近同尺度 Qwen3-8B AR；例如 AIME-24 为 69.6，LiveCodeBench-v6 为 45.7，相比 LLaDA-2.1-mini (16B) 分别高 26.3 和 15.3 分。
- 扩展：I-DLM-32B 对比 100B 级 LLaDA-2.x flash 仍有优势，例如 AIME-25 为 80.0，高于 LLaDA-2.1-flash 的 63.3。
- 效率：在高并发 serving 中，I-DLM 相对 LLaDA-2.1-mini 和 SDAR 有约 2.9--4.1x 的吞吐优势；系统消融显示 CUDA graph、stationary-batch decode loop、argmax proposals、paged-only attention 和 fused verify 累积带来 2.1--2.5x。

## 5. 写作故事线

论文的叙事从一个反常现象开始：DLM 理论上能并行生成，但在强 LLM 场景中既没有达到 AR 质量，也没有稳定转化成实用吞吐。作者没有先堆新 decoder，而是回到 AR 模型的基础结构，问为什么 AR 模型强。答案被组织成一个原则：causal masking + logit shifting 不只是 next-token prediction 的工程惯例，它让模型在同一个预测规则下同时学会生成和验证，因而具有 introspective consistency。

第二节把这个原则量化成 introspective acceptance rate。这里的关键转折是：作者不再只比较 benchmark accuracy，而是比较模型是否“认可自己刚才生成的 token”。这使得 LLaDA、SDAR、TiDAR、WeDLM 等工作被放到一个共同坐标系中：它们要么保留了某些扩散并行性但缺少严格验证，要么试图接入 AR serving stack 但仍依赖 confidence threshold 或额外 denoising/commit pass。

第三节据此提出 I-DLM。训练部分把 AR 的 causal/logit-shift 结构完整带入 diffusion conversion；推理部分用 ISD 把 proposal 和 verification 合并到一个节奏里；系统部分说明为什么 strict causal attention 让它天然适配 SGLang，而不是重新发明一套 DLM serving stack。

实验部分最后收束到两条证据链：第一，I-DLM 在质量上接近同尺度 AR 并超过已有 DLM，说明 introspective consistency 的确对应能力差距；第二，I-DLM 在高并发下吞吐随 TPF 更线性增长，说明它不是只在小 batch latency 上好看，而能落到服务系统经济性上。

## 6. 核心参考文献

- [1] Nie et al. 2025, *Large Language Diffusion Models*, arXiv:2502.09992 — LLaDA 系列的基础 DLM 代表，提供“扩散 LM 可扩展到 8B/更大规模”的正面背景，也是 I-DLM 要超越的主要质量基线。
- [2] Bie et al. 2026, *LLaDA2.1: Speeding Up Text Diffusion via Token Editing*, arXiv:2602.08676 — 与本文最接近的“修订/编辑能力”路线；本文认为它哲学上接近 introspection，但仍以 token editing 和 confidence-based decoding 为核心。
- [3] Cheng et al. 2025, *SDAR: A Synergistic Diffusion-AutoRegression Paradigm for Scalable Sequence Generation*, arXiv:2510.06303 — AR-to-DLM conversion 的强基线；本文批评其 block-causal attention、iterative denoising 与 KV commit pass 导致自洽性和服务效率不足。
- [4] Ye et al. 2025, *Dream 7B: Diffusion Large Language Models*, arXiv:2508.15487 — 引入/强调 logit shift 技术，说明本文的 logit-shifted causal training 不是凭空出现，而是把已有要素与 strict causal introspection 组合起来。
- [5] Tian et al. 2025, *From Next-Token to Next-Block: A Principled Adaptation Path for Diffusion LLMs*, arXiv:2512.06776 — NBDiff 代表 next-block adaptation，部分引入 causal prefix constraints；本文把它作为“朝 AR 结构靠近但未统一 verify-and-decode”的中间路线。
- [6] Liu et al. 2025, *WeDLM: Reconciling Diffusion Language Models with Standard Causal Attention for Fast Inference*, arXiv:2512.22737 — 同样强调 causal attention 与 KV cache 复用；本文与其差异在于 I-DLM 用 $p/q$ acceptance 给出 distributional guarantee，而 WeDLM 更依赖 streaming/confidence 机制。
- [7] Wu et al. 2025, *Fast-dLLM: Training-free Acceleration of Diffusion LLM by Enabling KV Cache and Parallel Decoding*, arXiv:2505.22618 — DLM inference 加速基线，说明只做 KV cache / parallel decoding 不一定解决训练自洽性和质量问题。
- [8] Leviathan et al. 2023, *Fast Inference from Transformers via Speculative Decoding*, ICML 2023 — $p/q$ acceptance 的理论来源；I-DLM 把 speculative decoding 的 accept/reject 规则内化到一个 DLM/AR hybrid 模型里。
- [9] Gloeckle et al. 2024, *Better & Faster Large Language Models via Multi-token Prediction*, arXiv:2404.19737 — 多 token prediction 是另一条向 AR 注入并行性的路线；I-DLM 可被理解为借助 diffusion-style MASK proposal 的 single-model MTP/verification 变体。
- [10] Zheng et al. 2024, *SGLang: Efficient Execution of Structured Language Model Programs*, NeurIPS 2024；Kwon et al. 2023, *Efficient Memory Management for Large Language Model Serving with PagedAttention*, arXiv:2309.06180 — 系统基础设施背景，解释为什么复用 AR serving stack 是本文效率 claim 的关键。

## 7. 批判性分析

**与 LLaDA / LLaDA2.1 的关系。** LLaDA (*Large Language Diffusion Models*, arXiv:2502.09992) 证明大规模 masked diffusion LM 可行，但标准 DLM 更偏“从 MASK 复原 token”，不是在同一 causal next-token 规则下验证 clean token。LLaDA2.1 (*Speeding Up Text Diffusion via Token Editing*, arXiv:2602.08676) 已经把“修订”作为核心，提出 token editing 和速度/质量模式；I-DLM 的新意在于把 revision 重新解释为 $p/q$ introspective acceptance，并让 clean positions 显式训练 causal anchor。也就是说，I-DLM 不是第一个看到 DLM 需要 revision 的工作，但它给了更接近 speculative decoding 的分布校正框架。

**与 SDAR / NBDiff / Dream 的关系。** SDAR (*A Synergistic Diffusion-AutoRegression Paradigm for Scalable Sequence Generation*, arXiv:2510.06303) 已经采用 AR-to-diffusion conversion，目标也是用少量适配成本获得 block-parallel generation。I-DLM 与 SDAR 的差别不是“是否从 AR 转换”，而是 attention mask、logit shift、clean-token supervision 和 decoding verification 是否统一。Dream 7B (*Diffusion Large Language Models*, arXiv:2508.15487) 已经使用 AR initialization 和 logit shift 一类技巧；NBDiff (*From Next-Token to Next-Block*, arXiv:2512.06776) 也把扩散 LM 往 next-block/causal prefix 方向推。本文贡献更像是把这些趋势合并成一个完整闭环：strict causal training + all-masked dense supervision + ISD 的 $p/q$ acceptance + serving stack。

**与 WeDLM / FastDLLM 的关系。** WeDLM (*Reconciling Diffusion Language Models with Standard Causal Attention for Fast Inference*, arXiv:2512.22737) 与 Fast-dLLM (*Training-free Acceleration of Diffusion LLM by Enabling KV Cache and Parallel Decoding*, arXiv:2505.22618) 都试图减少 DLM 与 AR inference engine 的错配。I-DLM 的强 claim 是：仅让 DLM 接近 causal attention 或支持 KV cache 不够，必须让 generation distribution $q$ 与 introspection distribution $p$ 在训练与推理中被同一机制约束。这个判断有说服力，但依赖作者自定义的 introspective acceptance rate 是否真能泛化为“质量代理指标”；论文目前主要在 IFEval 和下游 benchmark 上建立相关性，还缺少跨模型、跨采样温度、跨长上下文的系统相关性分析。

**与 speculative decoding、EAGLE-3、DFlash 的关系。** Speculative decoding (*Fast Inference from Transformers via Speculative Decoding*, ICML 2023) 已经给出 $p/q$ acceptance 的无偏校正规则；EAGLE-3 (*Scaling up Inference Acceleration of Large Language Models via Training-Time Test*, arXiv:2503.01840) 用额外 draft mechanism 加速 AR；DFlash (*Block Diffusion for Flash Speculative Decoding*, arXiv:2602.06036) 则直接用轻量 block diffusion drafter 做 lossless speculative decoding。I-DLM 的位置介于“新模型范式”和“自 speculative decoding”之间：full fine-tuned I-DLM 不是严格保持原 Qwen3 分布的 AR 加速器，而 R-ISD/gated LoRA 才是 bit-for-bit lossless 路线。批判性地看，论文在主质量表里主要报告 full I-DLM，而 lossless R-ISD 的质量由于构造上等于 AR 不成为主要表格焦点；因此“DLM 匹配 AR 质量”和“lossless acceleration”是两个相关但不同的 claim，阅读时不应混为一谈。

**与 multi-token prediction / gated LoRA 的关系。** Multi-token prediction (*Better & Faster Large Language Models via Multi-token Prediction*, arXiv:2404.19737) 早已提出让模型同时预测多个未来 token；Samragh et al. (*Your LLM Knows the Future: Uncovering Its Multi-Token Prediction Potential*, arXiv:2507.11851) 与本文所引的 gated sparse expert LoRA 接近 I-DLM 的 conditional LoRA 思路。I-DLM 的区别在于它不只是多头或多步预测，而是把 draft token 放入后续 causal anchor 里做 accept/reject。换言之，gated LoRA 本身不是本文独创核心，核心在于“MASK proposal 用 LoRA，clean verification 禁用 LoRA”与 strict causal mask 结合后能保持 base AR anchor 精确不变。

**与 vLLM / SGLang serving 的关系。** vLLM / PagedAttention (*Efficient Memory Management for Large Language Model Serving with PagedAttention*, arXiv:2309.06180) 和 SGLang 是本文系统 claim 的背景：高并发时吞吐取决于 KV cache、continuous batching、CUDA graph、attention kernel 和 scheduler 开销。I-DLM 很聪明的一点是把算法设计约束为 SGLang extend mode 能表达的 strict causal pattern，而不是事后给 block diffusion 补 serving 支持。但这也意味着它牺牲了部分传统 diffusion 的 bidirectional denoising 自由度；其“diffusion”属性更多体现在 MASK proposal 和 parallel refinement，而不是完整的多步全局 denoising。

**脆弱 claim。** 第一，“first DLM to match same-scale AR quality”依赖基线选择、thinking mode、采样配置与 benchmark 复现，尤其 Qwen3-8B / 32B 的结果会随 decoding recipe 变化。第二，I-DLM 的训练数据是 4.5B reasoning-response tokens，论文没有充分披露数据组成与去重，可能存在 benchmark-adjacent 数据风险。第三，系统吞吐优势强依赖 H100、SGLang、FlashInfer、CUDA graph 和长输出 2048 tokens 的测量设置；在短输出、低并发、非 NVIDIA 或带复杂 tool-use 的 workload 上，优势可能收窄。第四，argmax proposal 被称为不影响最终分布，是在严格 $p/q$ correction 成立时的理论结果；一旦实际实现存在 top-k/top-p 截断、数值近似、fused kernel 边界条件或 relaxed acceptance $\tau>0$，分布保证会变弱。

## 8. 方法细节

### 算法流程

**训练阶段。** 给定 clean sequence $x_0=(x_1,\ldots,x_L)$，构造 fully masked sequence $x_t=(\texttt{M},\ldots,\texttt{M})$，训练输入为 $[x_t \mid x_0]$。模型在 masked region 学 proposal distribution $q$，在 clean region 学 causal anchor distribution $p$。所有区域使用 strict causal attention，并采用 logit shift，让位置 $i$ 的 hidden state 预测 token $i+1$。

**ISD 推理阶段。**

1. 对 prompt 做标准 AR prefill。
2. append $N-1$ 或等价 block config 的 MASK tokens，单次 forward 产生一个 quality-guaranteed AR token $x_{L+1}$ 与后续 proposal tokens $x_{L+2:L+N}$。
3. 下一步 forward 把上一步 token 作为 clean/introspection positions，同时 append 新 MASK positions；clean positions 产生 $p_k$，MASK positions 产生新 proposal。
4. 对每个 proposal token 用 $p/q$ acceptance 判定；接受则提交，拒绝则从 corrected distribution 重采样并丢弃其后 proposal。
5. 若所有 proposal 都接受，则从最后 anchor 额外采样 bonus token，实现 $N+1$ 有效 stride。

### 关键公式

训练时 masked region 与 clean region 的 split loss 为：

$$
\mathcal{L}_{\text{mask}} = -\frac{1}{|\mathcal{S}_t|}\sum_{\ell \in \mathcal{S}_t} \log p_\theta(x_0^{\ell+1} \mid [x_t, x_0]_{\leq \ell}), \quad
\mathcal{L}_{\text{clean}} = -\frac{1}{|\mathcal{S}_0|}\sum_{\ell \in \mathcal{S}_0} \log p_\theta(x_0^{\ell+1} \mid [x_t, x_0]_{\leq \ell}).
$$

Auto-balanced loss（已据代码核对修订）：

$$
\mathcal{L} = \mathcal{L}_{\text{mask}} + \hat{s} \cdot \mathcal{L}_{\text{clean}}, \quad \hat{s} = \frac{\mathcal{L}_{\text{mask}}}{\mathcal{L}_{\text{clean}}},
$$

其中 $\hat{s}$ 在每个 training step 当作固定 scalar，不反传梯度。公开代码中的 `loss_auto_balance` 分支实现为 `task_loss + scale * clean_ce_loss`，`scale = task_loss.detach() / clean_ce_loss.detach()`；该分支没有再乘 YAML 里的 `ce_alpha`，而非 auto-balance 分支才是 `task_loss + ce_alpha * clean_ce_loss`。

ISD 的 acceptance probability：

$$
a_k = \min\left(1, \frac{p_k(x_k)}{q_k(x_k)}\right).
$$

拒绝时的 corrected resampling distribution：

$$
x_k \sim \text{normalize}(\max(0, p_k - q_k)).
$$

设 $p_k$ 为第 $k$ 个 strided token 的接受概率，$P_k=\prod_{j=1}^{k}p_j$，stride $N$ 下期望 tokens per forward pass 为：

$$
\text{TPF}_N = \frac{2 + P_1 + P_2 + \cdots + P_{N-2}}{2 - P_{N-1}}.
$$

appendix 在均匀接受率 $p$ 下给出 ISD 的 TPF 与 overhead：

$$
\text{TPF}_\text{ISD} = \frac{2 + p + p^2 + \cdots + p^{N-2}}{2 - p^{N-1}},
$$

$$
\text{OH}_\text{var} = \frac{3N - 1 - Np^{N-1}}{2 + p + \cdots + p^{N-2}},
$$

$$
\text{OH}_\text{fix} = \frac{(2N{-}1)(2 - p^{N-1})}{2 + p + \cdots + p^{N-2}}.
$$

对比 SDAR：

$$
\text{TPF}_\text{SDAR} = \frac{N}{\mathbb{E}[S \mid N] + 1}, \qquad
\text{OH}_\text{SDAR} = \mathbb{E}[S \mid N] + 1.
$$

对比 TiDAR：

$$
\text{TPF}_\text{TiDAR} = 1 + p + p^2 + \cdots + p^{N-1} = \frac{1 - p^N}{1 - p},
$$

$$
\text{OH}_\text{TiDAR} = \frac{N(N{+}1)}{1 + p + \cdots + p^{N-1}}.
$$

R-ISD 的 gated LoRA 每位置计算可写为：

$$
h_j \gets W x_j + \mathbf{1}_{[j \in \texttt{M}]} \cdot BA x_j.
$$

MASK proposal positions 使用 base+LoRA，clean/introspection positions 使用 base-only，因此 anchor $p$ 保持与 base AR forward 一致。

### 所用模型 / backbone / 数据流

- Backbone：Qwen3-8B 与 Qwen3-32B。
- I-DLM-8B：full fine-tuning，从 Qwen3-8B 转换。
- I-DLM-32B：从 Qwen3-32B 转换，训练描述中 32B 使用 LoRA rank 1024 的 stride $N=2$ 训练，并在 $N=4$ 评测。
- Lossless R-ISD：额外训练 LoRA adapters，8B rank 128；proposal 用 LoRA，verification 用 base-only。
- Serving：SGLang extend mode，FlashInfer attention，paged KV cache，continuous batching，CUDA graph capture；DLLM config 例如 `idlm_blockN4_config.yaml` 表示 stride $N=4$、`gen_block_size=4`、`block_size=2N-1=7`（已据代码核对修订）。

### 算力资源需求

论文明确给出的训练资源：

- 训练数据：论文训练使用 4.5B tokens 的 reasoning dataset responses；公开训练 YAML 默认挂的是 `open_thoughts3_sample`，README 和 YAML 均明确这是 demonstration sample，不是论文数据（已据代码核对修订）。
- 硬件：8 x NVIDIA H100 80GB SXM GPUs with NVLink。
- I-DLM-8B：2 epochs，stride curriculum：第一 epoch 为 $N=2$，第二 epoch 为 $N=3$；后续扩展到 $N=4$，主表使用 ISD $N=4$。
- I-DLM-32B：1 epoch，stride $N=2$，LoRA rank 1024；$N=4$ 用于评测。
- Hyperparameters：DeepSpeed ZeRO Stage 2，learning rate $1\times10^{-5}$，cosine decay，warmup ratio 0.03，per-device batch size 1，gradient accumulation 4，8 GPUs 上 effective batch size 32，maximum sequence length 4096，bf16。公开训练脚本只给 Qwen3-8B 的 b1/b2/b3 sample run，均为 8-GPU `torchrun`，sample YAML 中每个 block 配置为 1 epoch；论文中的完整 4.5B-token curriculum / 32B 训练配置没有以同等完整脚本释放（已据代码核对修订）。
- LoRA / R-ISD：rank 128，same data，learning rate 2e-4。公开仓库提供 LoRA serving 路径与 HuggingFace adapter 链接，但未看到 R-ISD LoRA 训练脚本（已据代码核对修订）。
- Serving hardware：质量/吞吐实验使用 H100；8B 为 TP=1，一台 server 1 GPU；32B 为 TP=2，一台 server 2 GPU；CUDA 12.9 + FlashInfer。

估算：若按 dense Transformer 训练 FLOPs 近似 $6 \times \text{params} \times \text{tokens}$，8B full fine-tuning 4.5B tokens 约为 $6 \times 8\text{B} \times 4.5\text{B} \approx 2.16\times10^{20}$ FLOPs。8 x H100 bf16 理论峰值取约 8 x 989 TFLOP/s，若实际利用率 25%--40%，对应墙钟大约 7.5--12 小时；若考虑长序列、ZeRO、I/O 和 all-masked doubled-sequence 开销，实际可能达到十几到二十余小时。论文没有给出真实 wall-clock，因此这里只能标为估算。

## 9. 实验

### 9.1 评测数据集

论文覆盖 15 个 benchmark，均启用 Qwen3 chat template 的 thinking mode，最大生成长度为 32,768 tokens。

- Knowledge and Reasoning：ARC-C (1,172), MMLU (14,042), MMLU-Pro (12,032), GPQA-Diamond (198), GPQA (448)。
- Math Reasoning：GSM8K (1,319), MATH-500 (500), MathBench (3,709), AIME-24 (30), AIME-25 (30)。
- Code Generation：HumanEval (164), MBPP sanitized (257), LiveCodeBench-v6 (175)。
- Instruction Following：IFEval (541)。
- 附加配置中还列出 TriviaQA (17,944)，但主表重点是上述 15 个。

吞吐评测使用 MBPP、MATH-500 和 LMSYS-Chat，concurrency $C \in \{1,2,4,8,16,32,48,64\}$，burst mode，固定输出长度 2048 tokens，warmup 5 requests。公开 `inference/eval/` 覆盖 ARC-C、MMLU、MMLU-Pro、GPQA、GSM8K、MATH-500、MathBench、AIME-24/25、HumanEval、MBPP、LiveCodeBench-v6、IFEval；未看到 TriviaQA eval 脚本（已据代码核对修订）。

### 9.2 评测指标

- 质量：accuracy / pass@1 / prompt-strict accuracy，结果为三次运行平均。数学题通过 `\boxed{}` 抽取最终答案，多选题抽取 ABCD，代码题抽取 Python code block 并用 10 秒 timeout 执行测试。
- 效率：request-level tokens per second 表示单请求 latency 侧速度；server-level tokens per second 表示整体吞吐。
- TPF：tokens per forward pass，用于衡量每次模型 forward 实际提交多少 token。
- Compute overhead：相对 AR 解码的总 query-token/FLOP 开销。
- Introspective acceptance rate：衡量 proposal distribution $q$ 是否会被 causal anchor distribution $p$ 接受。

### 9.3 主实验结果

**质量主表。** I-DLM-8B 在多个任务上接近 Qwen3-8B AR，并显著超过 DLM baseline：

- ARC-C：I-DLM-8B 95.8，Qwen3-8B 95.8，LLaDA-2.1-mini 90.2，SDAR-8B 91.9。
- MMLU：I-DLM-8B 82.4，Qwen3-8B 83.5，LLaDA-2.1-mini 74.5，SDAR-8B 78.6。
- MATH-500：I-DLM-8B 96.8，Qwen3-8B 95.8，LLaDA-2.1-mini 85.0，SDAR-8B 78.6。
- AIME-24：I-DLM-8B 69.6，Qwen3-8B 73.1，LLaDA-2.1-mini 43.3，SDAR-8B 10.0。
- AIME-25：I-DLM-8B 60.8，Qwen3-8B 65.4，LLaDA-2.1-mini 43.3，SDAR-8B 10.0。
- HumanEval：I-DLM-8B 93.3，Qwen3-8B 95.1，SDAR-8B 78.7。
- MBPP：I-DLM-8B 92.2，Qwen3-8B 93.4，SDAR-8B 72.0。
- LiveCodeBench-v6：I-DLM-8B 45.7，Qwen3-8B 50.3，LLaDA-2.1-mini 30.4，SDAR-8B 16.6。
- IFEval：I-DLM-8B 84.7，Qwen3-8B 84.7。

I-DLM-32B 相比 100B DLM baseline 也很强：AIME-25 80.0，高于 LLaDA-2.1-flash 63.3；LiveCodeBench-v6 57.1，高于 LLaDA-2.1-flash 45.4。

**扩展比较。** 在常见 DLM 报告指标上，I-DLM-8B 相比 NBDiff、Jacobi Forcing、WeDLM、LightningRL、TiDAR、DREAM、Fast-dLLM 等均更强。例如 HumanEval 93.3，高于 NBDiff 89.0、WeDLM 75.0、DREAM 57.9、Fast-dLLM 43.3；MBPP 92.2，高于 NBDiff 87.6、WeDLM 67.0、DREAM 58.8。

**吞吐。** 在 $C=16$--32 的典型部署规模，I-DLM 相对 LLaDA-2.1-mini 有 2.2--3.8x 更高吞吐，相对 SDAR 有 3.7--4.5x；$C=64$ 时每请求约 125 tok/s，总体约 2.9--4.1x 更高。对比 EAGLE-3，I-DLM 在 $C=1$ 到 $C=32$ 大多数设置更快；I-DLM-Lossless 在不少并发下也超过 EAGLE-3。

### 9.4 消融实验

**训练设计消融。** I-DLM 的 causal attention + logit shift + all-masked objective 对 long-horizon reasoning 尤其重要。去掉 causal/logit shift，退回 block diffusion 后：

- HumanEval：92.7 -> 60.3
- MBPP：92.8 -> 67.4
- MathBench：89.1 -> 71.6
- MMLU：82.4 -> 80.0

作者据此认为 introspective consistency 主要减少长链任务中的错误累积；知识类任务受影响较小。

**系统消融。** 从 naive implementation 开始累积优化：

- CUDA graph capture：+42%--76%，最大单项收益。
- Stationary-batch decode loop：+11%--21%，减少 host-side scheduling。
- Argmax proposals：+11%--15%，提高 acceptance rate 和 TPF。
- Paged-only attention：+10%--14%，用单 paged kernel 替代三段 cascade。
- Fused verify kernel：+1%--4%，收益较小但减少 kernel launch 和验证路径开销。
- 全部合计：约 2.1--2.5x throughput improvement。

**Stride size。** $N$ 控制并行性与质量：

- $N=2$：TPF 1.80，TPS 209.6，MATH-500 96.8，MBPP 93.4。
- $N=3$：TPF 2.48，TPS 281.9，MATH-500 95.8，MBPP 92.8。
- $N=4$：TPF 2.96，TPS 324.5，MATH-500 96.8，MBPP 92.2。
- $N=8$：TPF 4.01，TPS 445.1，MATH-500 94.6，MBPP 88.3。

**Relaxed acceptance。** 在 $N=4$ 下，放宽阈值 $\tau$ 会增加 TPF 但牺牲严格分布保证；HumanEval 从 $\tau=0$ 的 93.3 降到 $\tau=1$ 的 91.2，TPF 从 2.63 增到 2.73。

### 9.5 结果分析

**作者的分析。** 作者认为，I-DLM 的质量提升来自训练时把 generation pathway 与 introspection pathway 对齐：clean positions 学 causal anchor $p$，MASK positions 学 proposal $q$，并用 balanced loss 防止其中一路失衡。推理时，ISD 通过 $p/q$ acceptance 保证容易 token 并行前进，困难 token 自动回退到 AR-like 验证。系统方面，strict causal attention 让 I-DLM 直接进入 SGLang 的 extend/KV/cache/continuous batching 生态，避免 block diffusion 的同步与 commit pass。

**agent 的分析。** 我认为本文最强的地方不是单个技术点，而是把“训练目标、解码算法、serving implementation”统一约束到 causal/logit-shift 这一套结构上。这样做牺牲了一部分传统扩散的 bidirectional refinement 直觉，却换来了可验证的 acceptance、AR 兼容和工程可部署性。这个取舍很务实，也解释了为什么它在高并发吞吐上比只追求单请求低延迟的 DLM 更稳。

同时，论文的经验结论还需要后续复核。4.5B reasoning-response tokens 的数据来源不够透明，AIME、LCB、MATH 这类 benchmark 对训练数据污染很敏感；吞吐测量也高度依赖 H100/SGLang/FlashInfer/CUDA graph 和固定长输出。另一个值得关注的问题是 full I-DLM 与 R-ISD 的边界：full I-DLM 是一个经过转换的新模型，质量“接近 AR”；R-ISD 是一个 lossless acceleration 设计，输出等于 base AR。两者都重要，但证明的命题不同。

## 10. 代码实现观察

### 10.1 仓库基本情况

- 仓库：`https://github.com/Introspective-Diffusion/I-DLM`，本地 commit 为 `a23c1a12ef997c7f3ad616b25bcfb62db39ded68`。
- License：根目录 `LICENSE` 是 BSD 3-Clause License，copyright 标注为 Together AI。
- 权重：README 给出 HuggingFace collection，包含 `yifanyu/I-DLM-8B`、`yifanyu/I-DLM-32B`、`yifanyu/I-DLM-8B-lora-r128`。仓库内 `training/model/Qwen3-8B-b*-allmasked/` 有 config、tokenizer 和 `model.safetensors.index.json`，但没有实际 `.safetensors` 权重文件。
- 目录形态：训练端是 modified LlamaFactory + SDAR model；推理端是 bundled SGLang，新增 `sglang.srt.dllm`、`IDLMBlockN`、DLLM scheduler/mixin 和 eval scripts。
- 环境：训练 README 要求 Python 3.10、PyTorch >= 2.5.0、DeepSpeed、FlashAttention；推理 README / `setup_env.sh` 要求 Python 3.12、CUDA 12.9、PyTorch 2.9.1、FlashInfer，GPU compute capability >= 8.0，A100/H100 推荐。

### 10.2 方法实现核对

- 与论文一致：all-masked + clean 拼接确实在模型内部完成。`training/model/Qwen3-8B-b3-allmasked/modeling_sdar.py:247` 把 mask rate 固定为 1.0；`modeling_sdar.py:1109` 起构造 `concat_inputs_ids`，将 noisy 与 clean token 写入 2L 序列；`modeling_sdar.py:1127` 使用 block attention mask。关键几行是：

```python
# training/model/Qwen3-8B-b3-allmasked/modeling_sdar.py:247
p_mask_per_sample = torch.ones(num_samples_in_item, device=device)
# training/model/Qwen3-8B-b3-allmasked/modeling_sdar.py:1118
concat_inputs_ids[i][router_noisy_part[i]] = noisy_inputs_ids[i]
concat_inputs_ids[i][~router_noisy_part[i]] = inputs_ids[i]
```

- 与论文一致：strict causal / logit shift 是实做项。`modeling_sdar.py:344` 对 clean/x0 region 使用 token-level causal mask，`modeling_sdar.py:1197` 把 labels 左移一位，`modeling_sdar.py:1201` 又避免 packed samples 跨边界预测。
- 与论文一致：训练 loss 是 noisy/masked CE 加 clean-region AR CE。`training/llama_factory_sdar/src/llamafactory/train/sft/trainer.py:118` 调 SDAR forward 得到 `task_loss`；`trainer.py:135` 取 `hidden_states[:, seq_len:2*seq_len]` 过 `lm_head` 算 clean CE；`trainer.py:142` 实现 auto-balance。
- 需要修正的细节：README 中“CE loss on all non-padding positions / both noisy and clean”的说法偏宽。代码里 SDAR forward 的 `task_loss` 只对被 mask 的 noisy positions 取 loss，clean region 是 trainer 里额外算的 `clean_ce_loss`。最终目标仍符合论文的 `CE_noisy + alpha * CE_clean` / auto-balanced 设计。
- 需要修正的细节：`loss_auto_balance` 开启时，代码是 `task_loss + scale * clean_ce_loss`，`scale=task_loss/clean_ce_loss` 且 detached；没有再乘 `ce_alpha`。因此 sample b3 YAML 中的 `ce_alpha: 0.2` 在 auto-balance 分支不会改变 loss 权重。
- 与论文一致：公开 training scripts 使用 DeepSpeed ZeRO-2、bf16、8 GPU、per-device batch size 1、gradient accumulation 4、cutoff length 4096、cosine + 0.03 warmup。`run_train_b*-allmasked_idlm_sample.sh:48` 使用 `--nproc_per_node=8`；YAML 中 `deepspeed: examples/deepspeed/ds_z2_config.json`，`per_device_train_batch_size: 1`，`gradient_accumulation_steps: 4`，`bf16: true`。
- 论文未提及但代码中存在：训练脚本把 Triton / TorchInductor / CUDA cache 放到形如 `/dev/shm/${USER}/idlm_cache_<pid>` 的临时目录，并设置 `CC=gcc` 避免 conda compiler 找不到 `libcuda`；这说明实际训练对本机 CUDA toolchain 和共享内存缓存较敏感。
- 与论文一致：ISD 的 `blockN` 配置关系在 `inference/configs/*.yaml` 中明确，N=2/3/4/5/8/16 分别对应 `block_size=3/5/7/9/15/31`，`gen_block_size=N`，`use_spec_verify: true`。
- 需要澄清的细节：默认 `confidence_threshold: 0.0` 不是“跳过验证”。`IDLMBlockN` 里 `use_spec_verify` 默认 true，且 YAML 均开启；`confidence_threshold` 主要是旧式 confidence gate，真正的 lossless/relaxed 行为由 `verify_alpha`、`verify_num_specs`、`fast_verify` 等可选键控制。
- 与论文一致：p/q acceptance 和 corrected resampling 在 SGLang 集成中存在。`inference/sglang/sglang/srt/dllm/algorithm/idlm_blockN.py:684` 取 $p(x)$，`:687` 取 $q(x)$，`:696` 起计算 ratio，`:705` 起构造 `max(0, p-q)`。Triton fused path 在 `fused_verify_kernel.py:37`--`:40` 明确做 softmax、p/q gather、accept/reject 和 Gumbel-max correction。
- 论文未提及但代码中存在：非 greedy、非 fast verify 路径里，clean positions 走 top-k/top-p sampling，draft/spec positions 默认直接 argmax，并把 draft logits 留到下一步 fused verify 再 softmax。这对应论文系统消融里的 argmax proposals。
- 与论文一致：LoRA serving 的 conditional routing 存在。`IDLMBlockN._setup_conditional_lora` 对 verify/committed tokens 置 base-only，对 MASK/draft tokens 应用 LoRA；这与 R-ISD “anchor 保持 base AR，proposal 用 LoRA”一致。
- 与论文一致：SGLang integration 不是外部 wrapper，而是改入 server args、scheduler、model 和 algorithm。`server_args.py:4463` 添加 `--dllm-algorithm` / `--dllm-algorithm-config`；`scheduler.py` 新增 DLLM manager；`sdar.py` 将 attention 设为 `ENCODER_ONLY` 以便 DLLM forward batch 提供 mask。
- 论文未提及但代码中存在：DLLM inference 会强制/关闭若干 SGLang 功能。`server_args.py:2850` 在 CUDA graph 未禁用时强制 attention backend 为 FlashInfer；`:2856` 关闭 overlap schedule；`:2861` 关闭 radix cache；`:2884` 关闭 mixed chunked prefill；LoRA 只作为 algorithm-specific path 支持。

### 10.3 真实算力规模

- 公开训练脚本：`run_train_b1/b2/b3-allmasked_idlm_sample.sh` 都是单机 8 进程 `torchrun`。配套 YAML：`finetuning_type: full`，DeepSpeed ZeRO-2，`cutoff_len: 4096`，`per_device_train_batch_size: 1`，`gradient_accumulation_steps: 4`，`learning_rate: 1e-5`，`num_train_epochs: 1.0`，`bf16: true`。因此 sample run 的 effective batch size 是 $1 \times 4 \times 8 = 32$。
- 与论文声称的算力对照：论文报告的 8x H100、bf16、ZeRO-2、effective batch size 32 与公开脚本一致；但公开脚本是 sample dataset / sample run，不能直接复现实验里的 4.5B-token 训练，也没有完整释放 32B 与 R-ISD LoRA 训练脚本。
- 数据：`training/llama_factory_sdar/data/dataset_info.json` 只定义 `open_thoughts3_sample`，指向 `open-thoughts/OpenThoughts3-1.2M`；README/YAML 明确写明这不是 paper dataset。论文训练数据组成、去重、benchmark contamination 检查仍不可由代码验证。
- 推理：README 示例和 configs 默认面向 H100/A100 级别 GPU，单 8B server 常用 `tp-size 1`、`max-running-requests 32`、`attention-backend flashinfer`、`dtype bfloat16`；32B README 建议同样用 N=4 config，但主报告中论文吞吐部分的 32B TP=2 来自论文而非独立 launch script。

### 10.4 可复现性评价

- 评价：中等。推理复现性较高，训练复现性中等偏低。
- 有利因素：BSD 3-Clause license 清楚；模型权重和 LoRA adapter 已在 HuggingFace 发布；推理端有一键 `setup_env.sh`、OpenAI-compatible server 示例和覆盖主表多数任务的 eval scripts；核心训练和推理算法不是伪代码，占位较少。
- 主要缺口：论文的 4.5B-token 训练数据未公开；公开训练配置是 demonstration sample；没有看到完整 32B full/LoRA 训练脚本、官方 reproduction log 或固定 deterministic seed。普通研究者可以较容易跑通 inference/eval，但要复现论文训练结果需要自备数据、8x H100 级资源、WANDB/HF credentials 和较新的 CUDA/PyTorch/FlashInfer 环境。
- 已知环境坑：训练依赖 PyTorch flex attention、FlashAttention、DeepSpeed、Triton JIT 和可用 `nvcc`；脚本默认 `conda activate idlm`、需要 `/dev/shm` 缓存空间。推理要求 CUDA 12.9 / PyTorch 2.9.1 / FlashInfer 0.6.3，SGLang DLLM 模式会禁用 overlap schedule、radix cache、mixed chunked prefill，因此不能简单套用普通 SGLang 的所有部署选项。
- Eval scripts：脚本覆盖主表大部分 benchmark，但部分数据依赖本地路径或外部包。例如 MATH-500 优先读 `/data/.../test_prm800k_500.json`，不存在时才回退到 HuggingFace；MathBench 直接依赖 `/data/.../mathbench_v1/`；HumanEval/MBPP/LCB 会执行生成代码并有 timeout，复现时需要隔离运行环境。
