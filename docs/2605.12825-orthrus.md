# Orthrus：通过双视角扩散实现内存高效的并行 token 生成（Orthrus: Memory-Efficient Parallel Token Generation via Dual-View Diffusion）

## 1. 基本情况

- 题目（中 / 英）：Orthrus：通过双视角扩散实现内存高效的并行 token 生成 / Orthrus: Memory-Efficient Parallel Token Generation via Dual-View Diffusion
- 作者：Chien Van Nguyen, Chaitra Hegde, Van Cuong Pham, Ryan A. Rossi, Franck Dernoncourt, Thien Huu Nguyen
- 单位：University of Oregon, Google DeepMind, Adobe Research
- arXiv 编号：2605.12825v2
- 提交日期：v1 于 2026-05-12 提交；v2 于 2026-05-17 修订
- arXiv 链接：https://arxiv.org/abs/2605.12825v2
- PDF 链接：https://arxiv.org/pdf/2605.12825v2
- 代码仓库：https://github.com/chiennv2000/orthrus
- 主题分类：Machine Learning (cs.LG); Artificial Intelligence (cs.AI)

## 2. 核心贡献概述

- 中文版：Orthrus 在冻结的 autoregressive LLM 内部插入轻量 trainable diffusion attention module，让 AR view 负责高质量 context / KV cache 构建，让 diffusion view 一次并行预测多个 future tokens，再由同一个 frozen AR head 做 consensus verification，从而在不改变 base model 输出分布的前提下加速生成。它的关键卖点不是“diffusion 本身更准”，而是把 diffusion drafting 与 AR verification 做成同模型、共享 KV cache 的结构。
- English version（abstract）：`Orthrus guarantees lossless inference, delivering up to a 7.8x speedup with only an O(1) memory cache overhead and minimal parameter additions.`

## 3. 一句话精髓（写给外行）

Orthrus 像是在老练作家的脑子里装了一个会同时写好几句草稿的小助手，但每一句交稿前都由作家本人逐字点头，所以快了，却没有换掉原来的声音。

## 4. 学术贡献清单

### 4.1 动机的发现与阐明

作者抓住了 AR LLM 推理中的一个结构性不对称：prefill 阶段本来可以并行并产生高质量 KV cache，而 decode 阶段却必须逐 token 前进，主要受 memory bandwidth 和重复读取 KV cache 限制。已有 diffusion language models 能并行生成 token，但往往需要大规模训练、质量落后于同规模 AR model；已有 AR-to-diffusion adaptation 则会改变 base weights，使输出分布不可再严格等同于原 AR 模型。

这篇论文的动机不是重新证明 diffusion LM 可行，而是提出一个更窄也更工程化的问题：能否只把 parallel drafting 能力注入 frozen AR Transformer，同时保留 AR 模型原本的 exact predictive distribution 和 KV cache？

### 4.2 方法的创新

- 在每层 Transformer 中并列加入 trainable diffusion attention path；源码中新增的是 diffusion $Q/K/V/O$ projections 以及 diffusion q/k RMSNorm，而不是只有 $Q/K/V$，这也更接近论文所称 trainable parameters 约占 16% 的量级（已据代码核对修订）。
- AR view 和 diffusion view 共享历史 $(\mathbf{K}_{\text{AR}}, \mathbf{V}_{\text{AR}})$，diffusion view 只为固定大小的 parallel block 产生瞬时状态，因此历史 KV cache 没有线性级重复存储。
- 训练时用 dual-pass block masking：AR path 在 clean sequence 上产生 teacher distribution，diffusion path 看 clean prefix 加 anchor / `<mask>` block，用 forward KL 蒸馏 AR teacher 的完整 token distribution。
- 推理时采用 intra-model consensus：diffusion head 一次提出 $K$ 个候选，frozen AR head 在一个 verification pass 中计算 exact causal distribution，并只接受从左到右与 AR greedy / rejection sampling 对齐的最长前缀。

### 4.3 实验上的核心结论

- 在 Qwen3-1.7B / 4B / 8B 上，Orthrus 在 greedy 和 temperature sampling 设置下平均 speedup 约 $4.25\times$ 到 $5.36\times$，最大报告值为 Qwen3-8B 在 Pseudo2code 上的 $7.83\times$。
- 与 Dream-7B、LLaDA-1.5、SDAR、Fast-dLLM-v2、Mercury Coder、Gemini Diffusion 等 diffusion-based 模型相比，Orthrus-Qwen3-8B 在表中给出的数学和代码任务上保持 Qwen3-8B baseline 的 zero-shot 准确率，并有更高 MATH-500 / HumanEval / MBPP 数值。
- 与 EAGLE-3 和 DFlash 类 speculative decoding 对比，论文强调 Orthrus 的平均接受长度更高，并且在长 context 下因为不维护外部 drafter 的历史 KV cache，吞吐退化更小。
- 消融显示 $K=32$ 在 MATH-500 上使 TPF 达到 6.35；single-step block projection 比 two-step denoising 更快；KL soft distillation 与 hard CE 在 MATH-500 准确率相同，但 KL 的 TPF 更高。

## 5. 写作故事线

论文先从 AR LLM 的 decode bottleneck 入手：prefill 阶段的 self-attention 已经能并行处理 prompt，而后续生成必须一个 token 一个 token 地运行，导致 KV cache 被反复加载，硬件利用率低。然后作者引入 diffusion language models，承认它们在并行生成上天然有优势，但指出它们依赖 masked / denoising 条件独立近似，质量和训练成本都不理想。

在 §2，作者把矛盾形式化成两种概率建模范式：AR 通过 chain rule 保证每个 token 条件于完整历史，但采样串行；DLM 通过 block 内并行预测 amortize forward pass，但会偏离 strict causal dependency。这个数学铺垫是 §3 的转折点：Orthrus 不试图把 AR 模型“改造成” diffusion model，而是把两种 view 放进同一个 Transformer 里。

§3 的方法叙事围绕“分工”展开：frozen AR head 只负责构建可信历史表示和验证，trainable diffusion head 只负责并行提出候选。图 1 给出 dual-view architecture，图 2 把 training 与 inference 的 mask / verification 机制对齐起来：训练时 diffusion rows 对齐 AR rows，推理时 diffusion candidates 被 AR rows 验证。

§4 的实验先证明加速来自 TPF 和 wall-clock throughput，再把 Orthrus 放到两个邻近谱系里比较：一边是独立或 adapted diffusion LLM，另一边是 speculative decoding。作者借此强化核心 claim：Orthrus 的优势不是单点精度，而是“lossless + parallel + shared cache”三者同时成立。附录再补训练配方、objective ablation、memory scaling 与 limitation。

## 7. 批判性分析

与前作的关系：Orthrus 最像“同模型内嵌式 speculative decoding”，而不是纯粹的新 diffusion LM。DFlash（*DFlash: Block Diffusion for Flash Speculative Decoding*, arXiv:2602.06036）已经把 block diffusion drafter 与 target verifier 结合起来，并声称 single forward pass drafting 与 lossless acceleration；Orthrus 的关键差异是把 drafter 做成每层新增 diffusion attention path，让它直接复用 target model 的历史 KV cache。也就是说，Orthrus 的新意集中在 architectural integration 和 cache sharing，而 lossless guarantee 本身来自 AR verification / rejection sampling 框架。

相对于 Fast-dLLM-v2（*Fast-dLLM v2: Efficient Block-Diffusion LLM*, arXiv:2509.26328）和 SDAR（*SDAR: A Synergistic Diffusion-AutoRegression Paradigm for Scalable Sequence Generation*, arXiv:2510.06303），Orthrus 的路线更保守：不把 base model 变成 block diffusion model，而是冻结 base model，只训练新增 path。这个保守性带来精确分布对齐和低训练成本，但也带来明显边界：它不能超过 base AR model 的能力上限，所有生成质量 claim 都应理解为“保持 baseline，同时用更少 forward passes 达成”，而不是“diffusion view 自身更强”。

相对于 Block Diffusion（*Block Diffusion: Interpolating Between Autoregressive and Diffusion Language Models*, arXiv:2503.09573）、LLaDA（*Large Language Diffusion Models*, arXiv:2502.09992）、LLaDA-1.5（arXiv:2505.19223）和 Dream-7B（arXiv:2508.15487），Orthrus 放弃了 diffusion model 在 arbitrary-order generation、infilling、可控 denoising schedule 等方面的开放形态，而把 diffusion 限定为“未来 block 的 draft generator”。这让它非常适合 serving 加速，但不应被解读为证明 masked diffusion LM 全面替代 AR。

对 memory claim 需要谨慎。论文说 Orthrus 的历史 KV cache overhead 是 $O(1)$，并报告 $\Delta \approx 4.5$ MiB，这在其结构下合理：历史 cache 由 AR view 产生并复用，diffusion path 只处理固定 $K=32$ 的临时 block。但“优于所有 speculative decoding 的内存开销”依赖具体实现；某些 speculative methods 的 draft model 可以更小、量化或不保留完整长历史 cache。因此，最稳妥的表述是 Orthrus 在“需要长上下文且 drafter 也要保留随 context 增长状态”的场景中更有结构优势。

对 speed claim 也要区分指标。表 1 的 TPF 是 hardware-agnostic proxy，而 speedup 列来自相对 AR baseline 的测量；最大 $7.83\times$ 出现在 Pseudo2code / Qwen3-8B / greedy 设置。由于每个 generation cycle 至少包含 diffusion projection 和 AR verification 两次 forward，实际吞吐会强依赖 GPU kernel、batch size、context length、accepted prefix length 和 serving framework。论文虽用 FlexAttention + FlashAttention-4 做训练 mask，但推理端是否在不同部署栈中维持同等收益，仍需要独立复现。

比较表中的 diffusion baselines 也有时间和版本问题。Orthrus 是 2026-05-17 的 v2，而 DFlash 在 arXiv 页面显示 2026-05-28 有 v2 camera-ready，Fast-dLLM-v2、SDAR 等在 2025 年后持续更新；这些工作发展很快，单表对比可能受 checkpoint、decoding setting、benchmark contamination 和 prompt protocol 影响。尤其 Gemini Diffusion 是 Google DeepMind 的实验 demo 页面，非同一开放 checkpoint / 同一评测脚本，因此只能作为速度质量现象参照，不宜当作严格 apples-to-apples baseline。

脆弱 claim：论文说“strictly lossless generation”在 greedy decoding 下由式 (5) 的 exact match 很清楚；temperature $T>0$ 时作者提到 exact rejection sampling，但正文没有完整展开 proposal distribution、acceptance probability、top-k/top-p/penalty 等实际采样细节。如果生产环境使用 nucleus sampling、repetition penalty、grammar constraints 或 tool-call constraints，需要确认 verification 逻辑仍等价于目标分布。

## 8. 方法细节

### 算法流程

1. 从预训练 AR LLM（实验中为 Qwen3-1.7B / 4B / 8B）出发，冻结原 backbone 与 AR attention path。
2. 在每个 Transformer layer 旁路加入 diffusion attention module；源码实际包含 diffusion $Q/K/V/O$ projection 与 diffusion q/k norm，训练时应只更新这些 diffusion path 参数，原 backbone 保持冻结（已据代码核对修订）。
3. Prefill 阶段：AR view 对 prompt / clean context 做标准 causal forward pass，产生高质量历史 $(\mathbf{K}_{\text{AR}}, \mathbf{V}_{\text{AR}})$。
4. 训练阶段：对长度 $L$ 的 clean sequence 随机采样 $B$ 个 anchor positions，每个 anchor 后取长度 $K$ 的 block，保留第一个 token 作为 anchor，其余替换为 `<mask>`；diffusion view 在特殊 block mask 下预测该 block 内未来 token distribution，并用 forward KL 对齐 frozen AR teacher。
5. 推理阶段：diffusion view 以当前 token 为 anchor，加上 $K-1$ 个 `<mask>`，单次 forward 得到 $K$ 个候选 token。
6. Verification 阶段：frozen AR head 用一个 forward pass 对完整候选 block 计算 causal target probabilities；从左到右接受与 AR greedy prediction 一致的最长前缀。若第 $j$ 个 token 不一致，则提交前 $j-1$ 个候选和 AR correction token，并截断/同步 KV cache。
7. 对 stochastic decoding，论文称使用 exact rejection sampling 对齐目标分布。

### 关键公式

AR 语言模型目标：

$$
\mathcal{L}_{\text{AR}}(\theta) = -\mathbb{E}_{\mathbf{x} \sim \mathcal{D}} \left[ \sum_{i=1}^N \log p_{\theta}(x_i \mid \mathbf{x}_{<i}) \right]
$$

Masked Diffusion LM 目标：

$$
\mathcal{L}_{\text{MDM}}(\phi) = -\mathbb{E}_{\mathbf{x} \sim \mathcal{D}, t, \mathbf{y}^t} \left[ \sum_{k \in \mathcal{M}} \log p_{\phi}(y_k^0 \mid \mathbf{c}, \mathbf{y}^t) \right]
$$

DLM 的并行近似：

$$
p_{\text{DLM}}(\mathbf{y}^0 \mid \mathbf{c}) \approx \prod_{k \in \mathcal{M}} p_{\phi}(y_k^0 \mid \mathbf{c}, \mathbf{y}^t)
$$

Orthrus diffusion view attention：

$$
\mathbf{O}_{\text{diff}} = \text{Softmax}\!\left(\frac{\mathbf{Q}_{\text{diff}}\,[\mathbf{K}_{\text{AR}} \,\|\, \mathbf{K}_{\text{diff}}]^{\top}}{\sqrt{d_{\text{head}}}}\right) [\mathbf{V}_{\text{AR}} \,\|\, \mathbf{V}_{\text{diff}}]
$$

其中 $[\cdot\|\cdot]$ 表示沿 sequence axis 拼接，$\mathbf{O}_{\text{diff}} \in \mathbb{R}^{K \times d_{\text{head}}}$。

Block corruption：

$$
\tilde{y}_{b,k} = \begin{cases} x_{a_b} & k = 1 \\ \texttt{<mask>} & k = 2, \dots, K \end{cases}
$$

Diffusion view 的 training mask：

$$
\mathbf{M}_{\text{diff}}[q, k] =
\underbrace{\mathbf{1}[k < L] \cdot \mathbf{1}[k \leq a_b - 1]}_{\text{causal AR context}}
\;\;|\;\;
\underbrace{\mathbf{1}[k \geq L] \cdot \mathbf{1}\!\left[\lfloor q/K \rfloor = \lfloor (k-L)/K \rfloor\right]}_{\text{bidirectional within block}}
$$

Orthrus 训练目标：

$$
\mathcal{L}_{\text{Orthrus}} = \mathbb{E}_{\mathbf{x}, \{a_b\}} \left[ \sum_{b=1}^{B} \sum_{k=1}^{K} D_{\text{KL}}\!\left( p_{\text{AR}}(\cdot \mid \mathbf{x}_{\leq a_b + k - 1}) \;\|\; p_{\text{diff}}(\cdot \mid \mathbf{x}_{< a_b}, \tilde{\mathbf{y}}_b) \right) \right]
$$

Greedy consensus：

$$
\hat{y}_k = \arg\max_{v \in \mathcal{V}} p_{\text{AR}}(v \mid \mathbf{x}_{\le t}, \hat{\mathbf{y}}_{1:k-1})
$$

Efficiency metric：

$$
\text{TPF} = \frac{\text{Total Generated Tokens}}{\text{Total Forward Passes}}
$$

### 所用模型 / backbone / 数据流

- Backbone：Qwen3-1.7B、Qwen3-4B、Qwen3-8B。论文表 2 中 Orthrus-Qwen3-8B 参数量为 8.7B，说明新增 diffusion path 带来约 0.7B 参数。
- 数据源：Nemotron-Post-Training-Dataset-v2，按 Chat / Math / Code 三域 1:1:1 平衡采样。
- 数据规模：600K examples，经 sequence packing 得到 471,952 packed instances，总计 0.96B tokens。
- 输入长度：max sequence length $L=2048$。
- Block 设置：parallel block size $K=32$；每个 training sequence 采样 256 anchor blocks。
- 训练信号：目标输出由 frozen AR head 直接生成，diffusion head 最小化 forward KL；公开仓库只保留了 diffusion training mask / forward 接口，没有提供 KL loss 与训练 loop 的实现（已据代码核对修订）。
- 推理数据流：prompt -> AR prefill cache -> diffusion block projection -> AR verification -> accepted prefix append to shared cache -> 下一轮。

### 算力资源需求

论文明确给出的训练配置如下；公开仓库未提供训练脚本、FSDP 配置或数据处理脚本，因此这些数字仍主要来自论文而非可直接运行的代码配置（已据代码核对修订）：

- 硬件：single 8xH200 GPU node。
- 训练时长：作者在贡献里称 under 24 hours。
- 并行和内存：PyTorch FSDP-2；micro-batch size = 1 per device；gradient accumulation steps = 16；global batch size = 128。
- 精度：bfloat16。
- 优化：cosine learning rate scheduler；peak learning rate $2 \times 10^{-4}$；warmup ratio 0.05；gradient clipping 1.0。
- Attention kernels：FlexAttention + FlashAttention-4 backend，用于自定义 training masks。
- 训练轮数：2 epochs。
- 训练 token：0.96B packed tokens。
- 可训练参数：约 16% total parameters。

估算：以 471,952 packed instances、global batch 128、2 epochs 计算，优化步数约为 $471{,}952 \times 2 / 128 \approx 7{,}374$ steps。若 8B backbone 新增约 0.7B trainable parameters，但每步仍需要 frozen backbone forward 产生 teacher / cache，则训练计算量显著高于只训练一个小 adapter；不过无需 full-model backward，成本主要来自 frozen teacher forward、diffusion path forward/backward 与大规模 masked block attention。

推理内存：论文附录报告 peak GPU memory penalty 约 100 MiB，KV cache overhead 为固定 $\Delta \approx 4.5$ MiB，随 context length 近似 $O(1)$。该结论依赖 $K=32$ 固定 block 和历史 AR cache 共享。

## 9. 实验

### 9.1 评测数据集

- GSM8K：小学数学 word problems，检验基础 arithmetic reasoning。
- MATH-500：从 MATH benchmark 取出的高难数学题，检验复杂多步推理。
- AIME24 / AIME25：美国数学邀请赛题目，检验竞赛级数学推理。
- HumanEval：Python 函数生成，使用单元测试评估 functional correctness。
- MBPP：基础编程题，检验短代码生成能力。
- Pseudo2code：把 procedural / pseudo-code 描述转成代码，强调结构化与长程序生成。
- LiveCodeBench-v5：较新的代码生成评测，意在降低训练污染风险。
- 长上下文吞吐实验：论文图 4 右侧比较不同 context length 到 40K 时的 end-to-end throughput。

### 9.2 评测指标

- Accuracy / pass rate：数学题和代码题的最终正确率，数学通常看答案匹配，代码看测试通过。
- TPF（Effective Tokens Per Forward Pass）：总生成 token 数除以总 forward pass 数，用于隔离算法级并行效率。
- Relative speedup：相对于 sequential AR baseline 的吞吐加速倍数。
- Average Acceptance Length：每次 verification 平均接受的 token 数，衡量 draft 与 target distribution 的对齐程度。
- Tokens per second / wall-clock throughput：真实部署相关吞吐，论文在 MATH-500 speed 图和 long-context 图中使用。
- Memory footprint / KV cache overhead：peak GPU memory 和随 context length 增长的 KV cache 额外开销。

### 9.3 主实验结果

效率表中，Orthrus 在 8B 模型上表现最强：

- Qwen3-8B greedy 平均 TPF = 5.39，平均 speedup = $5.36\times$。
- Qwen3-8B temperature $T=1$ 平均 TPF = 4.43，平均 speedup = $5.02\times$。
- 最大 speedup 为 Pseudo2code / Qwen3-8B / greedy：TPF = 7.51，speedup = $7.83\times$。
- MATH-500 / Qwen3-8B / greedy：TPF = 6.35，speedup = $5.95\times$；论文图中称 wall-clock 约 $6\times$。

SOTA diffusion 对比表中，Orthrus-Qwen3-8B 报告：

- GSM8K 96.0，MATH-500 86.2，AIME-24 28.3，AIME-25 23.3，HumanEval 95.1，MBPP 93.4。
- 对比 SDAR-Qwen3-8B：GSM8K 91.7，MATH-500 78.6，AIME-24 10.0，AIME-25 10.0，HumanEval 78.7，MBPP 72.0。
- 对比 Fast-dLLM-v2：GSM8K 83.7，MATH-500 61.5，HumanEval 43.3，MBPP 28.2。
- 对比 Gemini Diffusion：AIME-25 20.0，HumanEval 89.6，MBPP 76.0。

Speculative decoding 对比中，论文报告 MATH-500 上 Orthrus average acceptance length = 11.7，高于 DFlash 7.9 和 EAGLE-3 3.5。长上下文吞吐图声称 Orthrus 到 40K context 仍保持接近峰值吞吐，而 DFlash 随 context 增长明显退化。

### 9.4 消融实验

- Block size $K$：在 MATH-500 / Orthrus-Qwen3-8B 上，增大 $K$ 提高 TPF；$K=32$ 时 TPF = 6.35，较 $K=4$ 有 3.6x throughput multiplier，且作者称 forward-pass latency 基本不随 $K$ 增长。
- Single-step vs multi-step denoising：multi-step 需要 2 passes，TPF = 3.53；Orthrus single-step 只需 1 个 diffusion projection pass，TPF = 6.35。作者据此选择 single-step。
- KL vs CE 训练目标：在 MATH-500 上两者 accuracy 都为 86.2%，但 hard CE 的 TPF = 5.86，forward KL 的 TPF = 6.35。作者解释为 hard labels 学到的是数据表面 token，而 KL 更好贴合 AR teacher 的 full predictive distribution，verification rejection 更少。
- Memory scaling：附录图报告 KV cache overhead 固定约 4.5 MiB，不随 sequence length 线性增长。

### 9.5 结果分析

作者的分析：Orthrus 的性能来自结构性解耦。AR backbone 冻结，所以最终准确率继承 base Qwen3，而不会像 converted diffusion models 那样出现 conditional drift。Diffusion head 只负责提出候选，不承担最终分布正确性；AR head 的 intra-model consensus 保证输出与 base model 一致。共享 KV cache 让它比外部 speculative drafter 更适合长上下文，因为没有随 context 增长的第二份历史 cache。

agent 的分析：实验最有说服力的部分是“保持 base accuracy + 提高 TPF + 内存不随 context 线性增加”的组合，尤其适合 serving 端追求 latency / throughput 的场景。但论文的质量优势几乎完全绑定到 base model，本质上是一个高接受率 drafter 训练问题；如果 base model 较弱、任务分布偏离训练三域、或采样策略复杂，diffusion head 的 acceptance length 可能下降，实际 speedup 会随之下降。另一个待复核点是与 DFlash/EAGLE-3 的 memory 和 long-context 对比应在同一 serving stack、相同 quantization、相同 batch / context / decoding policy 下独立验证。

## 10. 代码实现观察

### 10.1 仓库基本情况

- 仓库地址：https://github.com/chiennv2000/orthrus；本地核对 commit 为 `0ac515d4c88118704114001cf7ddf20ff7f0898c`（`feat: add mlx backend support`）。
- 顶层文件很少，主要是 `README.md`、`pyproject.toml`、`LICENSE` 与 `src/{configuration.py,model.py,model_mlx.py}`；没有 `train.py`、`scripts/`、`configs/`、数据处理脚本或 eval runner。
- License 信号不一致：仓库根目录 `LICENSE` 是 MIT；`pyproject.toml:16` 写 `Apache-2.0`；Hugging Face model card / API 对 `chiennv/Orthrus-Qwen3-8B` 标为 `cc-by-4.0`。代码使用时应优先让作者澄清源码、权重和模型卡三者各自的 license。
- README 提供 1.7B / 4B / 8B 三个 Hugging Face checkpoint 链接；HF API 抽查 8B checkpoint 显示 `model.safetensors`、`modeling_orthrus.py`、`auto_map: {"AutoModelForCausalLM": "modeling_orthrus.OrthrusLM"}`，因此 PyTorch quickstart 依赖 `trust_remote_code=True`。
- README 明确写 vLLM / SGLang native integration 是 “Coming soon”，本仓库当前没有 vLLM 或 SGLang backend。

### 10.2 方法实现核对

- 与论文一致：`src/model.py:80-182` 继承 Qwen3 attention 并加入 diffusion attention path；AR pass 走原 `Qwen3Attention`，diffusion pass 使用 `q_proj_diff/k_proj_diff/v_proj_diff/o_proj_diff` 与 diffusion q/k norm。`src/model.py:145-154` 将历史 AR cache 的 key/value 与当前 diffusion block 的 key/value 拼接，符合 shared KV cache 的设计。
- 与论文一致：training mask 的核心规则在 `src/model.py:47-77`。AR cache 部分通过 `kv_idx <= causal_limit[b, q_idx]` 限制每个 diffusion query 可看的 causal prefix；diffusion block 内通过 `q_block_id == kv_block_id` 允许同一 block 内互看。训练态 diffusion pass 强制要求 `causal_limit` 和 `ar_seq_len`，并用 FlexAttention block mask（`src/model.py:285-296`）。
- 与论文一致：PyTorch `generate` 的 `use_diffusion_mode=True` 路径先用 AR prefill 生成第一个 token，再构造 `[anchor, <mask>, ...]` diffusion block（`src/model.py:447-477`），随后用同一模型 AR pass verify proposed block（`src/model.py:481-497`），最后 `past_key_values.crop(start_idx)` 截断未接受 token 的 cache（`src/model.py:531-532`）。
- 与论文一致：greedy consensus 按最长相同前缀接受，随机采样路径也有接受率 `min(1, p/q)` 与 residual distribution 采样（`src/model.py:498-513`），对应论文中的 stochastic verification 思路。
- 与论文不一致或需要修正原报告表述：公开源码新增的不只是 diffusion $Q/K/V$，还包括 `o_proj_diff` 和 diffusion q/k RMSNorm；因此“只训练 Q/K/V”应理解为“只训练 diffusion attention path”，严格说源码层面是 $Q/K/V/O$ plus q/k norm。
- 与论文不一致或未完全覆盖：`src/model_mlx.py` 的 Apple MLX 后端实现了双 path、共享 cache 和 greedy-style verification，但 temperature > 0 时只是分别采样 diffusion / AR token 后比较相等（`src/model_mlx.py:214-233`），没有 PyTorch 路径中的 rejection sampling / residual distribution，因此不能把 PyTorch 的 stochastic lossless 保证直接外推到 MLX 后端。
- 论文未提及但代码中存在的工程细节：PyTorch diffusion training attention 使用 `torch.compile(flex_attention, dynamic=False)`，并在 kernel options 中设置 `BACKEND: "FLASH"` 和 sparse block size（`src/model.py:31-45`）。推理支持 `attn_implementation="sdpa" | "eager" | "flash_attention_2" | "flash_attention_4"`，README 安装项还提示可装 `flash-attn-4[cu13]`。
- 论文未提及但代码中存在的限制：当前 PyTorch `generate` 分配 `output_ids` 为 shape `(1, max_length + block_size)`（`src/model.py:422`），自定义 diffusion generation 路径实际按 batch size 1 写；普通 `GenerationMixin.generate` 只在 `use_diffusion_mode=False` 时使用。
- 未在公开代码中找到：forward KL loss、teacher logits 生成、参数冻结策略、FSDP-2 包装、gradient accumulation、训练数据 packing、Nemotron 三域采样、benchmark/eval 脚本。

### 10.3 真实算力规模

- 从公开代码能确认的推理配置：HF 8B `config.json` 抽查显示 `block_size=32`、`mask_token_id=151669`、`dtype=bfloat16`、`max_position_embeddings=40960`、`num_hidden_layers=36`、`num_attention_heads=32`、`num_key_value_heads=8`，与论文的 $K=32$、长上下文到 40K、Qwen3-8B 设置一致。
- 从公开代码能确认的依赖和硬件假设：`pyproject.toml` 要求 `torch>=2.10.0`、`transformers>=5.8.0`、`accelerate>=1.13.0`；README 额外要求 `flash-attn`，并提示支持 `flash-attn-4[cu13]`。这意味着 CUDA / PyTorch / Transformers 版本都偏新，训练侧 FlexAttention + Flash backend 对环境要求较高。
- 论文中的训练规模（8xH200、micro-batch 1、grad accumulation 16、global batch 128、2 epochs、bf16、FSDP-2、warmup 0.05、grad clip 1.0、约 0.96B tokens）未在仓库配置文件中落地；公开仓库只能作为推理和模型结构证据，不能作为训练算力规模的可执行证据。
- HF API 抽查 8B checkpoint 的 safetensors 参数量约 9.70B BF16 parameters；这高于 README 表格中的 base “Qwen3-8.0B”，符合额外 diffusion path 带来的参数增加。

### 10.4 可复现性评价

- 推理可复现性：中。README 给出 Hugging Face checkpoint 和 PyTorch quickstart，`use_diffusion_mode=True`、streamer 和 MLX Apple Silicon 示例都在公开代码中有对应实现；但依赖版本很新，且需要 `trust_remote_code=True` 与合适的 attention backend。
- 训练可复现性：低。仓库没有训练脚本、loss 实现、数据下载/packing 脚本、FSDP 配置、seed、训练日志或 checkpoint 转换说明；普通研究者无法仅凭仓库复现论文中的 forward KL distillation 和 8xH200 训练流程。
- 实验可复现性：低到中。模型权重公开，理论上可复跑部分下游评测和速度测试；但仓库没有官方 eval harness、prompt protocol、batch/context/quantization 设置、long-context throughput 脚本，也没有 DFlash/EAGLE-3 对照实验脚本。
- 部署成熟度：当前更像 “HF remote-code inference release + MLX preview”。README 的 vLLM/SGLang 仍是 coming soon，因此生产 serving 需要等待专门 backend 或自行移植 cache crop / verification 逻辑。
